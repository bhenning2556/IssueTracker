Issue Tracker SRS

Author: Benjamin Henning, developer



1. Introduction

1.1. Purpose

This System Requirements Specification (SRS) documentation will describe the functionality, requirements and constraints of the Issue Tracker application. This document will cover system features, functional requirements, non-functional requirements, external interface requirements, design and implementation requirements and constraints, assumptions, and dependencies.

1.2. Intended Audience

The stakeholders for this document are as follows:

The developer, Benjamin Henning (me).
The developer will be in charge of writing the SRS ahead of the product development, and will be the sole contributor to changes.

Employers, recruiters and clients.
This document will be publicly available, but the intended audience is anyone wishing to see how this project was created and developed, mainly employers, recruiters and clients. The document will aid in understanding the design process, intended features and functionality, and use cases of the application.

1.3. Intended Use

This document will be used in the following ways:

  1. To develop a project timeline with weekly sprints and deadlines.
  2. To lay out requirements, features and functionality of the system to guide development.
  3. To describe use cases of the application for end users that will aid in testing and demonstration.

1.4. Scope

The Issue Tracker application is intended to be used by development teams, project managers, and individuals to track software bugs and issues. The application will give users functionality to view, add, edit, delete, and mark solved bugs and issues. The application will allow an admin to give users roles that give varying levels of permissions to access and manage bug tracking.

1.5. Definitions

Term Definition
User Someone who interacts with the Issue Tracking application
Admin System administrator who is given specific permission for managing and controlling the system
CRUD Create, Read, Update and Delete. Describes how data is managed and viewed in a database-driven application.
MVC Model, View, Controller - A popular software design pattern used to build web applications.

2. Overall Description

The issue tracker software will be a solution for Project Managers and Development Teams to track bugs and issues in their systems. The application will have CRUD functionality, meaning users and admins will be able to easily manage and store bug data in a relational database. The application be built using the MVC design pattern, so the code and file structure will be familiar to any third parties who view or work on the application after the development period.

2.1. User needs

The intended audience for the application is developers and people with basic technical knowledge. However, the application's functionality should be affordable to the point that non-technical users can intuitively use basic features.

2.2. Assumptions and Dependencies

It is assumed that this software will be used for its intended purpose, which is to track bugs and issues for a piece of software or a system. It is assumed that users will have a basic knowledge of bug tracking applications and how they function.

3. System Features and Requirements

3.1. Functional Requirements

  • Landing page
    • Login/register
      • Users will be able to choose whether to login to an existing account or register for a new account.
      • If the user selects register, they will be able to sign up for a new account and provide an email and password.
      • If the user selects login, they will be prompted to enter an email and password.
      • Users will be able select a "forgot your password" option to have a password reset sent to their email.
  • Dashboard
    • Users will be able to select which organization to view projects for.
    • Users will be able to select a project to view.
    • Users will be able to see statistics about projects in their organization.
  • Projects
    • Users will be able to perform CRUD operations on active issues. Creating or deleting issues has limited permissions.
  • User Profile
    • Users will be able to edit their profile details, such as their profile image, display name, email address, and password.
  • Organizations
    • Users will be able to create a new organization, or join an existing one. Upon creating a new organization, the user will be assigned the Owner role by default.
    • Users with the role of Owner will be able to invite users to join their organization.
    • Users with the role of Owner will be able to assign or change roles of users in their organization.
    • Users with the role of Owner will be able to create new projects for their organization.

3.2. Non-Functional Requirements

  • Visual Design and Affordance
    • The web application should have an aesthetically pleasing design.
    • The web application should be responsive, so that it can be used on any screen size.
    • The navigation elements should indicate to the user their current location in the application (ex. the 'Dashboard' element should be highlighted if the user is viewing the dashboard).
  • Readability and Accessibility
    • All text should be easily distinguishable from background elements and typeface should be readable.
    • All colors on the web application should comply with AA contrast standards.
    • Different content should be contained in distinguishable sections.
  • Security
    • Application functionality should only be accessible to users with their assigned role's permission level (ex. A user that has been assigned a 'developer' role should not have access to functionality that only user's with the 'manager' role have permission for).
    • Account information, such as emails and passwords, should not be stored as plain text in the database.

3.3. System Requirements

  • A device with an internet connection.
  • A compatible browser.
